\section{課題1・2の概要}

\subsection{背景と当事者性・切実性}
教科書は、授業中には必要で、学期が終わると余りやすい。問題は、この時間差の両側で学生が別々に負担を抱えることである。購入する学生は新品価格や配送費を負担し、手放す学生は梱包・発送の手間を避けるうちに、使わない教科書を本棚へ残してしまう。

私たちにも、学期の初めに複数冊をまとめて購入し、授業が終わったあと、まだ使える教科書を譲る相手が見つからないまま保管した経験がある。翌年度の履修者へ渡せればよいとは思っても、誰が同じ授業を取るのか、その人がどのキャンパスにいるのかは分からない。買う側だったときの負担と、売る側になったときの行き詰まりは、同じ教科書の前後に生じていた。

既存のフリマアプリや古本屋でも売買はできる。ただし、大学の授業で使われる特定の版・科目の教科書を探す場面では、次の二点が残る。
\begin{enumerate}
    \item 配送、梱包、販売手数料などが、少額の中古教科書取引に対して相対的に重い。
    \item 書名だけでは授業との対応が分かりにくく、同じ大学・キャンパスの出品へ絞り込めない。
\end{enumerate}

同じキャンパスに通う学生同士なら、配送そのものを省ける。そこで「OU Textbook」では、大阪大学の学生が教科書を検索し、価格と受け渡し日時・場所に合意したうえで、キャンパス内で直接受け渡す流れを一つのWebアプリケーションにまとめた。もっとも、対面取引にすれば問題が消えるわけではない。待ち合わせの不履行と、当事者でない利用者の介入をどう抑えるかが、設計上の焦点となった。

\subsection{普遍性と社会性}
教科書の購入と処分は、特定の学部や学年だけに生じる問題ではない。豊中・吹田・箕面の各キャンパスで、履修科目が変わるたびに繰り返される。とりわけ新入生にとっては、教科書代が入学時の諸費用と同時に発生するため、中古教科書を授業単位で探せる意義は大きい。

再利用には資源面の効果もあるが、本提案の直接的な価値は、まだ使える教科書と、それを必要とする次の履修者を学内で結び付ける点にある。対象を大阪大学の学生へ限定することで利用者数は狭まる一方、キャンパス、授業分野、受け渡し可能性という文脈は濃くなる。この限定が、汎用サービスとの差になる。

\subsection{情報技術で解く範囲}
送料をなくすだけなら、SNSで相手を募って手渡せばよい。そうしたかった、とは思う。しかし、投稿が流れる場所では検索条件、合意内容、取引履歴が分散し、待ち合わせの直前に誰が何へ同意したのかを追いにくい。そこで、次の三つをシステム要件とした。
\begin{itemize}
    \item \textbf{見つける}: 書名・著者名の部分一致検索に、キャンパス、カテゴリ、価格順の条件を組み合わせる。
    \item \textbf{当事者を限定する}: \texttt{@ecs.osaka-u.ac.jp} のメールアドレスを登録条件とし、取引操作は出品者、購入希望者、購入者の権限に応じて制御する。
    \item \textbf{合意を残す}: チャット、価格提示、受け渡し提案、双方の完了確認、相互評価を段階別のデータとして保存する。
\end{itemize}

信用スコアは最後の要件を補助する。良い評価は$+10$点、悪い評価と予定確定後のキャンセルは$-10$点、無断不参加の報告は$-30$点とし、同じ評価や報告が重複適用されないよう制約を設けた。スコアだけでトラブルを防げるわけではないが、過去の取引結果を次の相手が判断する材料にはできる。

費用をかけずに再現できることも条件とした。Webフレームワークには3条項BSDライセンスで公開されているDjangoを採用した\cite{django-license}。データベースと画像保存は、ローカルではSQLiteとファイルシステムだけで動作する。Supabaseもオープンソースの構成要素から成り、Dockerを用いたセルフホストが可能である\cite{supabase-selfhost}。したがって、外部の有償サービスを契約しなくても、PC上で認証を含む構成を再現できる。ただし、公開サーバーの維持費まで常にゼロになるわけではない。

\section{設計・作成したシステム・ソフトウェアの説明}

ローカルではSQLiteを使い、\texttt{DATABASE\_URL}が設定された環境では外部データベースへ接続する。画像は、所定の環境変数がある場合はSupabase Storageへ、ない場合はローカルファイルシステムへ保存する。

認証サービスを分離しても、取引データの整合性はDjango側に残る。価格提示の承諾、受け渡しの確定、完了確認、評価の適用は、状態と操作者の両方を検査してから更新する。第三者がURLを推測しても取引を確定できないことが必要だった。

\subsection{データモデルと取引状態}
実装した中心的なモデルは、次の八つである。
\begin{description}
    \item[\texttt{UserProfile}]
    表示名、学部・学科、学年、Supabase利用者ID、信用スコアを保持する。スコアに応じて「要注意」「レギュラー」「トラスト」「エキスパート」の表示を切り替える。

    \item[\texttt{Book}]
    書名、著者、価格、カテゴリ、キャンパス、状態、画像、出品者と購入者を保持する。状態は\texttt{available}、\texttt{in\_progress}、\texttt{sold}、\texttt{withdrawn}の四つである。

    \item[\texttt{Favorite} / \texttt{Message}]
    お気に入りは利用者と書籍の組を一意に保存する。メッセージは書籍、送信者、受信者、本文、送信時刻を保存し、書籍ごとの相談履歴を構成する。

    \item[\texttt{TradeOffer}]
    出品者から特定の購入希望者への価格提示を保存する。一冊に対して成立済みの提示が複数残らないよう、条件付き一意制約を置いた。

    \item[\texttt{HandoffProposal}]
    成立した価格提示に対する受け渡し日時・場所と、売り手・買い手双方の完了確認時刻を保存する。

    \item[\texttt{Evaluation} / \texttt{CancellationLog}]
    相互評価とキャンセル・無断不参加の報告を保存する。一意制約と適用済みフラグにより、信用スコアの二重加算・減算を防ぐ。
\end{description}

取引は、相談を始めただけでは成立しない。出品者が価格を提示し、対象の購入希望者が承諾した時点で、書籍は\texttt{available}から\texttt{in\_progress}へ移る。受け渡し予定時刻を過ぎた後、双方が完了を確認し、双方の評価がそろうと\texttt{sold}になる。途中でキャンセルされた場合は購入者との紐付けを外し、再び\texttt{available}へ戻す。

\subsection{動作環境とローカル構築手順}
依存バージョンは\texttt{requirements.txt}で固定または範囲指定している。主な構成はDjango 6.0.6、SQLite（ローカル）、Supabase Authである。本番用設定の\texttt{render.yaml}ではPython 3.14.3を指定し、\texttt{DATABASE\_URL}による外部データベース接続、WhiteNoiseによる静的ファイル配信、Supabase Storageによる画像保存に対応した。

Windows PowerShellでソースコードを取得済みの場合、次の手順で起動できる。
\begin{lstlisting}
cd C:\path\to\pbl

python -m venv .venv
.\.venv\Scripts\Activate.ps1
python -m pip install -r trading_text\requirements.txt

$env:SUPABASE_URL="https://xxxx.supabase.co"
$env:SUPABASE_ANON_KEY="your-supabase-anon-key"

cd trading_text
python manage.py migrate
python manage.py check
python manage.py runserver
\end{lstlisting}

画像をSupabase Storageへ保存する場合は、これらに加えて\texttt{SUPABASE\_STORAGE\_BUCKET}と\texttt{SUPABASE\_SECRET\_KEY}を設定する。秘密鍵はソースコードへ書かず、環境変数として渡す。

\subsection{最も苦労した点と次に苦労した点}
\subsubsection{最も苦労した点: Supabase AuthとDjango標準認証の統合}
Supabase Authの利用者とDjangoの利用者は、同じ人物を表していても別の識別子と状態を持つ。認証成功後にメールアドレスだけで結び付ければ簡単に見えるが、表示名の初期化、既存利用者との同期、Supabase利用者IDの保存、Djangoセッションの開始までを一貫させなければならない。

そこで、Supabaseの認証APIを呼ぶ処理を\texttt{supabase\_auth.py}へ分離し、ビュー側の\texttt{sync\_supabase\_user}でローカル利用者とプロフィールを同期した。認証サービスのエラーをフォームへ返しつつ、成功時だけDjangoのログイン状態を作る境界の調整に最も時間を要した。

\subsubsection{次に苦労した点: アップロード画像の検査と保存先の切替}
拡張子だけを検査しても、ファイルの実体は判別できない。実装では5MBの上限を設け、先頭16バイトをJPEG、PNG、GIF、WebPの候補シグネチャと照合してから保存するようにした。また、ローカルと本番で保存先が変わっても\texttt{Book.image}の扱いを変えずに済むよう、DjangoのStorageインターフェースを用いた。

ただし、先頭部分の照合は完全な画像デコードではない。特にRIFF形式はWebP以外にも使われるため、現在の検査だけで悪意あるファイルを排除できるとは言い切れない。この限界は未解決である。

\subsection{未解決の部分と今後の課題}
実装を終えても、対面取引の不確実さは残った。限られた期間で、検索、価格合意、受け渡し、完了確認、評価という取引の中心経路と、第三者の操作を拒否する権限制御を優先したためである。外部サービスとの継続的な通信が必要な機能と、運用者の判断基準が必要な機能は、短いローカル試験だけでは安全性を確かめられず、実装範囲から外した。

優先度の高い順に、次の改善が必要である。
\begin{enumerate}
    \item \textbf{認証セッションの更新と失効処理}: 現在はSupabaseから得たアクセストークンとリフレッシュトークンをセッションへ保存するが、期限切れ時の自動更新と、Supabase側で失効した利用者をローカルセッションへ反映する処理が不足している。原因は、登録・ログインの成立を先に検証し、時間経過や外部での失効を含む長時間試験まで実施できなかったことである。
    \item \textbf{画像の完全な検証}: Pillow等で画像を実際にデコードし、形式、画素数、破損の有無を検査したうえで再エンコードする必要がある。現在の検査は追加ライブラリを用いず先頭バイトだけで判定する方針を採ったため、RIFFコンテナの内部形式まで識別できなかった。
    \item \textbf{チャット通知の即時化}: 現在は画面の再読込で新着メッセージを取得する。Django Channels等によるリアルタイム更新、またはメール通知が必要である。通常のHTTPによる取引機能を先に完成させた結果、WebSocket接続の維持と切断時の再接続を検証する期間を確保できなかった。
    \item \textbf{受け渡し場所の選択支援}: 現在は場所を自由記述する。学内の明るく人通りのある候補地点を選択式で示し、必要に応じて地図上で確認できる形が望ましい。候補地点の選定には各キャンパスの現地確認が必要であり、APIを接続するだけでは安全な場所を保証できないため、自由記述にとどめた。
    \item \textbf{紛争対応と運用設計}: 学内メール認証と信用スコアは本人確認や補償の代替ではない。虚偽の無断不参加報告、金銭トラブル、不適切な出品に対する異議申立てと管理者対応を定める必要がある。これはプログラムだけでは決められず、証拠の扱い、管理者の権限、個人情報の保存期間について合意できる運用主体が必要になる。
\end{enumerate}

\section{結論}
余っている教科書と、次の学期にそれを必要とする学生は、同じキャンパスにいる。それでも取引が成立しにくかったのは、相手を見つける情報と、価格・日時・場所への合意が別々の手段に散らばっていたからである。

OU Textbookは、学内メール認証、教科書検索、チャット、価格提示、受け渡し合意、完了確認、評価を一つの状態遷移として実装した。これにより、配送を使わない教科書取引を、単なる呼びかけから記録可能な手続きへ変えられた。一方で、認証失効、画像の完全検査、紛争対応は残っている。学期が変わるたびに安心して使える仕組みになるかどうかは、この三点を運用まで含めて詰められるかにかかっている。
